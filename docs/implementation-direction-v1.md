# Data Agent V1 实现方向草案

## 1. 当前目标

基于上传文件做数据分析，不接数据库。

支持的核心输入形式：

- Excel
- CSV

支持的核心使用形式：

- 生成结构化分析报告
- 基于当前数据进行自然语言问答

## 2. 相比原 PDF 方案的关键调整

原方案以广告数据分析为主路径，这个方向保留；但 V1 的系统设计不应被广告场景写死。

当前确认的调整如下：

1. 数据来源先限定为本地上传文件，不做数据库接入。
2. 产品能力要支持“多分析智能体”扩展，而不是只做一个固定广告 Agent。
3. 广告数据分析是首个重点场景，但底层能力需要可复用到其他业务数据分析场景。
4. 在正式分析前，系统必须先做澄清、确认，再生成分析计划，再执行分析。
5. 对于月报类场景，输出不是普通聊天回复，而是结构强、排版明确、可预览的报告。

## 3. 建议的产品定位

不要把它定义成“一个广告数据分析聊天机器人”，而应该定义成：

一个支持创建多个分析智能体的文件型 Data Agent 工作台。

其中每个分析智能体本质上是一套可配置的分析上下文，至少包含：

- 智能体名称
- 场景描述
- 适用数据类型说明
- 字段和指标解释
- 分析范围与排除规则
- 常见任务模板
- 报告模板
- 输出风格约束

例如：

- 广告数据智能体
- 销售数据智能体
- 用户运营数据智能体

## 4. V1 推荐的产品结构

V1 推荐采用“智能体工作台”模式，而不是“上传文件后直接聊天”模式。

### 4.1 左侧

- 智能体列表
- 新建智能体入口
- 当前会话历史

### 4.2 中间

- 对话区
- 文件上传区
- 澄清问题
- 分析计划确认
- 执行状态

### 4.3 右侧

- 报告预览面板
- 指标卡
- 图表
- 结构化结论
- 导出入口

这和原 PDF 中“左对话、右报告”的主结构一致，但扩展成了“多智能体 + 多会话”的平台形态。

## 5. 核心交互流程

### 5.1 通用问答流程

1. 用户选择某个分析智能体
2. 上传数据文件
3. 输入问题
4. 系统先做数据解析与语义识别
5. 系统提出澄清问题或展示待确认项
6. 用户确认
7. 系统生成分析计划
8. 用户确认计划
9. 系统执行分析
10. 左侧继续对话，右侧展示结果卡、图表或报告

### 5.2 月报生成流程

1. 用户选择广告数据智能体
2. 上传数据
3. 发起“生成月报”
4. 系统先检查字段映射、时间范围、指标口径、过滤规则是否完整
5. 系统展示待确认项
6. 用户点击确认按钮
7. 系统生成月报分析计划
8. 用户确认计划
9. 系统生成结构化月报，并在右侧面板持续写入

## 6. 为什么建议做“多智能体”，但不要过度多 Agent 化

你的需求里提到“可以有多个分析智能体，可以支持新建”。这个方向是对的。

但实现上建议区分：

- 产品层：多智能体
- 系统层：少数固定运行角色

也就是说，用户看到的是多个可配置的业务智能体；但系统内部不一定要真的搞很多彼此独立、长期记忆复杂协作的 Agent。

V1 更稳妥的做法是：

- 一个可配置的业务智能体定义
- 一个固定的 Planner
- 一个固定的 Analyst
- 一个固定的 Report Builder

这样更容易控质量，也更方便后续扩展。

## 7. V1 必须单独设计的“智能体定义”

你提到新建智能体时要添加“一些描述”或“类似规范的东西”。这个非常关键。

建议不要只让用户填一个名字和一句描述，而是让新建智能体时配置一份结构化定义。

建议字段如下：

- 智能体名称
- 场景说明
- 适用数据说明
- 常见分析目标
- 核心指标定义
- 核心维度定义
- 字段别名映射
- 字段选择规则
- 数据清洗规则
- 分析排除规则
- 月报模板说明
- 回答风格说明

其中最重要的是后 6 项，因为它们直接决定 Agent 是否真的懂你的业务。

## 8. 广告数据场景下最关键的能力不是问答，而是“数据语义层”

你补充的信息里，真正复杂的点不是聊天本身，而是：

- 字段很多
- 字段名相似
- 同义字段多
- 描述性字段不规则
- 分析前有大量条件判断
- 某些字段需要优先判断或先排除
- 同一需求背后有较长的业务解释链条

所以 V1 的关键模块不应该只是 LLM 对话，而应该是一个可配置的数据语义层。

这个语义层建议至少包含：

- 字段字典
- 字段别名
- 指标定义
- 维度定义
- 默认过滤条件
- 排除条件
- 值识别规则
- 分析优先级规则
- 报告模板配置

## 9. 建议把复杂规则做成“可维护配置”，而不是全部写死在 Prompt

像你说的这些规则：

- 先看有没有备注
- 某些 URL 前缀才纳入分析
- 某些字段只能二选一
- 某些维度名很像但口径不同

如果全靠 Prompt 硬写，后面会非常难维护。

V1 建议拆成两层：

### 9.1 业务配置层

用结构化配置保存：

- 字段映射
- 过滤规则
- 指标口径
- 模板定义
- 澄清项模板

### 9.2 LLM 推理层

基于配置去做：

- 问题理解
- 需要澄清的点识别
- 分析计划生成
- 分析结论生成
- 报告文案生成

## 10. 推荐的系统模块设计

### 10.1 Agent Registry

负责管理多个分析智能体的定义。

### 10.2 File Intake

负责文件上传、文件切换、文件解析、字段识别、类型推断。

### 10.3 Semantic Config

负责字段字典、指标定义、字段别名、清洗规则、过滤规则、模板规则。

### 10.4 Clarification Engine

负责在执行前输出：

- 我对需求的理解
- 当前缺失的信息
- 存在歧义的字段或指标
- 建议你确认的选项

### 10.5 Planner

负责根据当前数据、智能体配置和用户需求，产出结构化分析计划。

### 10.6 Analysis Executor

负责执行数据筛选、聚合、计算、趋势分析、异常识别。

### 10.7 Report Builder

负责将分析结果渲染为：

- 指标卡
- 图表
- 结构化报告

### 10.8 Conversation Layer

负责聊天记录、按钮交互、追问上下文和状态同步。

## 11. UI 层面的明确建议

你提到确认项希望做成聊天气泡中的按钮，这个建议应该纳入 V1。

建议的交互元素：

- 澄清问题卡片
- 单选确认按钮
- 多选确认按钮
- “按此计划执行”按钮
- “重新生成计划”按钮
- 执行中的 loading / streaming 动画
- 右侧报告实时增量写入

这部分不只是视觉细节，它直接影响用户是否能理解 Agent 当前在做什么。

## 12. V1 范围建议

### 12.1 必做

- 多智能体管理
- 文件上传（Excel / CSV）
- 字段识别
- 澄清问题
- 分析计划确认
- 自然语言问答
- 月报生成
- 右侧报告面板

### 12.2 可以先简化

- 多文件联合分析
- 高级归因模型
- 长期记忆
- 自动数据库接入
- 复杂权限体系
- 高级可视化编排

## 13. 当前最值得先定下来的三件事

在动手做之前，最应该先定的是：

1. 智能体定义结构
2. 数据语义配置结构
3. 月报生成的标准骨架

如果这三件事没定，后面越做越容易散。

## 14. 当前我的判断

这件事的技术核心不是“做一个会聊天的 Agent”，而是：

做一个基于上传文件、可配置语义规则、先澄清再规划再执行、同时支持问答和结构化报告输出的数据分析工作台。

广告数据只是第一个重点场景，不应该成为系统架构上的硬编码限制。
