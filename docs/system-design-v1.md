# System Design V1

## 1. 范围

本文件用于把当前已确认方向收敛成一版可实现的 V1 系统方案。

相关文档：

- `implementation-direction-v1.md`
- `agent-definition-spec-v1.md`
- `semantic-config-spec-v1.md`
- `monthly-report-template-v1.md`

## 2. V1 产品定义

V1 不是一个单纯的聊天机器人，也不是一个通用 BI 工具。

V1 应定义为：

一个基于上传文件的数据分析工作台，支持创建多个分析智能体，并通过“澄清 -> 计划 -> 执行 -> 输出”的流程完成问答和结构化报告生成。

## 3. V1 业务边界

### 包含

- 多智能体
- Excel / CSV 上传
- 字段识别
- 语义配置
- 澄清确认
- 分析计划确认
- 自然语言问答
- 月报生成
- 左对话右报告
- 单月与跨月月报

### 不包含

- 数据库接入
- 多租户权限体系
- 高级因果归因
- 长期记忆系统
- 复杂后台配置平台
- 通用多文件任意拼接分析

说明：

这里“不包含通用多文件任意拼接分析”，但广告月报场景允许受控的多月 raw data 输入，用于跨月对比。

## 4. 核心架构思路

建议系统采用“一个工作台 + 多业务智能体配置 + 少数固定运行角色”的模式。

### 4.1 用户可见层

- 智能体选择 / 新建
- 文件上传
- 对话与澄清
- 分析计划确认
- 报告预览

### 4.2 系统运行层

- Agent Registry
- File Intake
- Semantic Resolver
- Clarification Engine
- Planner
- Analysis Executor
- Report Builder
- Conversation State Manager

### 4.3 配置层

- Agent Definition
- Semantic Config
- Report Templates

## 5. 推荐的模块拆分

### 5.1 Agent Registry

职责：

- 管理多个智能体定义
- 切换当前智能体
- 提供预置智能体模板

输入：

- 用户选择的智能体

输出：

- 当前智能体配置

### 5.2 File Intake

职责：

- 接收 Excel / CSV
- 解析文件
- 识别 sheet 与表头
- 基础类型推断
- 产出字段识别结果
- 在受控场景下识别多月 raw data 并做月份归档

输出：

- dataset metadata
- sample rows
- detected headers
- candidate mappings

### 5.3 Semantic Resolver

职责：

- 读取语义配置
- 完成字段映射
- 找出冲突和歧义
- 输出标准化字段语义
- 产出标准费用字段和按指标生效的排除规则

### 5.4 Clarification Engine

职责：

- 汇总缺失项、歧义项、风险项
- 生成可点击的确认卡片
- 在正式规划前完成关键信息对齐

### 5.5 Planner

职责：

- 根据用户意图生成结构化分析计划
- 区分问答任务与月报任务
- 把计划返回给用户确认

建议输出格式：

```json
{
  "task_type": "monthly_report",
  "summary": "基于 3 月广告投放数据生成月报",
  "steps": [
    "校验时间范围与字段映射",
    "计算核心指标",
    "生成趋势分析",
    "做关键维度拆解",
    "输出报告"
  ],
  "assumptions": [],
  "needs_confirmation": []
}
```

### 5.6 Analysis Executor

职责：

- 数据清洗
- 过滤
- 聚合计算
- 趋势分析
- 维度拆解
- 异常检测
- 文本字段语义预处理
- 派生指标重算
- 缺失数据和不可监测样本处理

说明：

V1 可以先走代码执行型路线，不一定把所有分析都交给 LLM。LLM 更适合理解需求、生成计划和撰写解释文本。

其中：

- `备注`
- `adslot`

这类自然语言字段建议先做规则化语义分类，再进入正式分析。

另外：

- CPM
- CPC
- 点击率
- 到达率
- 各类成本

都建议由 Executor 基于当前过滤条件、当前聚合粒度、当前有效样本重新计算，而不是直接采用源表中的现成值。

### 5.7 Report Builder

职责：

- 把执行结果拼成结果卡、图表和结构化报告
- 管理右侧报告面板内容
- 支持流式增量写入

### 5.8 Conversation State Manager

职责：

- 管理会话状态
- 记录当前数据集
- 记录已确认项
- 记录当前计划
- 记录执行状态

## 6. 推荐的任务状态机

建议把一次分析任务显式建模为状态机。

```text
idle
-> file_uploaded
-> schema_detected
-> clarification_pending
-> clarified
-> plan_pending_confirmation
-> plan_confirmed
-> executing
-> result_ready
```

好处：

- 前后端状态更清楚
- 用户为什么被要求确认更容易解释
- 右侧报告何时开始写入更容易控制

## 7. 两类核心任务

### 7.1 问答任务

特点：

- 更轻量
- 更即时
- 输出通常是指标卡、图表、摘要

### 7.2 月报任务

特点：

- 有固定结构
- 必须做更多确认
- 输出偏报告化

结论：

这两类任务共享底层模块，但 Planner 和 Report Builder 需要区分任务类型。

## 8. 前端建议的主界面结构

### 左侧

- 智能体列表
- 会话列表

### 中间

- 对话流
- 上传入口
- 澄清卡片
- 计划确认卡片
- 执行进度

### 右侧

- 报告预览
- 指标卡
- 图表
- 分析摘要

## 9. 澄清卡片建议

你要求“确认项放在聊天气泡里并用按钮点击”，我认同，而且建议作为主流程标准化。

建议卡片类型：

- 字段冲突确认卡
- 时间范围确认卡
- 指标口径确认卡
- 默认规则启用确认卡
- 执行计划确认卡

## 10. 为什么不建议一开始就做“很多内部 Agent”

虽然产品层面叫多智能体，但系统层面 V1 不建议做太复杂的多 Agent 编排。

原因：

- 调试困难
- 状态同步复杂
- 结果责任边界不清
- 你当前真正需要的是配置能力和流程透明，不是炫技式 agent orchestration

V1 更建议：

- 业务智能体配置化
- 运行角色固定化

## 11. 建议的落地顺序

### 第一阶段

- 固定配置结构
- 跑通上传、字段识别、澄清、计划

### 第二阶段

- 跑通问答任务
- 跑通月报任务

### 第三阶段

- 做更完整的右侧报告面板
- 做预置广告数据智能体模板

## 12. 当前建议的技术原则

- 配置优先于 prompt 堆砌
- 显式确认优先于模型猜测
- 代码执行优先于纯文本分析
- 状态机优先于隐式流程
- 报告结构优先于一次性长文本

## 13. 当前结论

V1 的正确起点不是先写一个聊天框，而是先把“配置结构 + 状态机 + 主流程”定清楚。
