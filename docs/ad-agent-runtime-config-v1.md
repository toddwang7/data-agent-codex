# Ad Agent Runtime Config V1

## 1. 目的

这份文档把“广告数据智能体”的方案进一步收敛成接近可执行的配置结构。

重点不是解释概念，而是回答：

- 平台未来到底要配什么
- 后端执行到底依赖什么
- 前端确认卡片怎么被驱动

## 2. 当前已经落到配置里的四类关键能力

### 2.1 字段映射

已在 `config/agents/ad-analysis/semantic-config.json` 中加入 `header_mapping_hints`。

作用：

- 把 Excel 真实表头直接映射到标准字段
- 让固定 schema 的广告 rawdata 在上传后能快速进入标准化流程

### 2.2 报告月份判定

已加入 `report_period_resolution`。

原因：

- 文件名月份和行内 `投放时间` 区间不一定一致
- 月报不能只靠原始行里的日期区间自动决定

当前建议优先级：

1. 用户显式确认
2. 文件名月份标签
3. 主导时间区间推断

其中月报任务必须确认。

### 2.3 指标重算策略

已加入 `source_metric_trust_policy`。

核心规则：

- 源表里已有的率和成本列只作为参考
- 最终分析一律按当前样本和当前口径重算

当前被列为“不直接作为最终结果源”的字段包括：

- `点击率`
- `到达率`
- `有效访客成本`
- `高意向访客成本`
- `留资率`
- `留资成本`

### 2.4 澄清卡片流

已加入 `clarification_flow`。

当前顺序：

1. 确认报告月份
2. 确认点位分析字段
3. 确认费用口径
4. 确认 FREE 点位处理方式
5. 确认特殊样本处理方式

这意味着前端不需要硬编码每一步，而是可以按配置驱动聊天气泡卡片。

## 3. 当前这版配置意味着什么

现在这套广告智能体配置，已经不是“写一段 prompt”。

它已经具备了 4 层结构：

- 数据识别层
- 业务规则层
- 分析执行层
- 交互确认层

这也是为什么以后换成别的数据智能体时，平台必须支持结构化配置，而不是只支持写描述。

## 4. 下一步最自然的工程动作

如果继续往下做，最合理的顺序是：

1. 按这套配置定义一份正式 JSON Schema
2. 写一个上传后跑字段识别和澄清卡片的最小后端流程
3. 再接分析执行和右侧报告渲染

## 5. 当前结论

到这一步，广告数据智能体已经可以被视为一套“平台可配置的业务分析模板”，而不是临时人工说明。
